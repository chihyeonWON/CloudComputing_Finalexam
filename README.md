클라우드 컴퓨팅 환경/스택 구조

CCD : Cloud Computing Service Client Device : 클라우드 컴퓨팅 서비스 단말

CN : Cloud Computing Network : 클라우드 네트워크

CSP : Cloud Computing Service Provider : 클라우드컴퓨팅 서비스 제공자

클라우드컴퓨팅 생태계 구성 5가지

클라우드 컴퓨팅 서비스 제공자 : CSP 클라우드 컴퓨팅 인프라 위에 클라우드 플랫폼을 구성, IaaS, PaaS, SaaS 등 다양한 서비스를 제공

클라우드 컴퓨팅 서비스 중개자 : CSB Cloud Computing Service Brokerage
다수의 이종 클라우드 서비스를 연계하여 사용자에게 제공
서비스 간 통합 및 최적화를 통한 신규 서비스와 가치를 창출
사용자 요구사항 기반의 최적 서비스를 선택, 배치하는 기능을 제공

클라우드 컴퓨팅 네트워크 : CN Cloud Computing Network 
클라우드 제공자와 클라우드 단말을 연결하는 각종 유무선 망

클라우드 컴퓨팅 서비스 단말 : Cloud Computing Service Client Device
클라우드 서비스를 이용하는 수단, 예 : 스마트폰, 태플릿 PC, PC와 노트북

클라우드 컴퓨팅 보안 : CS : Cloud Computing Security
클라우드를 안전하게 이용할 수 있는 기능을 제공

클라우드 컴퓨팅 구성 기술 (5가지)

1. 가상화 : 물리적인 IT 리소스를 시스템 또는 프로그램 형태로 운영

2. 분산처리(Distributed Processing) : 대규모 서버 환경에서 대용량 데이터를 네트워크에서 분산 처리
모든 사용자에게 전체 시스템이 아닌 하나인 것처럼 보이게 함
내부적으로는 상호 분산되어 수행되는 시스템

3.오픈 인터페이스(Open Interface) : 인터넷을 통해 서비스를 이용, 서비스 간에 정보 공유를 할 수 있는 인터페이스

4. 오케스트레이션(Orchestration) : CSP가 이용자의 서비스 요청으로부터 제공까지 자동화 하는 기술
컴퓨터 시스템과 응용, 서비스의 자동화된 구성, 관리, 조정을 의미

5. 보안 및 프라이버시 보호(Security and privacy protection) : 외부 컴퓨팅 자원에 기업/개인의 민감정보를 안전
하게 보호하는 기술

NIST CCRA의 5개 행위자 (클라우드 컴퓨팅의 주요 행위자, 행위자들의 행위와 기능을 식별)

클라우드 소비자 : 클라우드 제공자와 비즈니스 관계를 유지하고 서비스를 사용하는 개인 또는 조직
클라우드 제공자 : 이해 관계자가 사용할 수 있는 서비스를 만들 책임이 있는 사람, 조직 또는 단체
클라우드 감사 : 클라우드 서비스, 정보시스템 운영, 클라우드에 대한 성능 및 보안에 대한 독립적인 평가를 수행하는 파트
클라우드 브로커 : 클라우드 서비스의 사용, 성능 및 제공을 관리하고 클라우드 제공업체와 클라우드 소비자 간의 관계를 조정
클라우드 캐리어 : 클라우드 제공업체에서 소비자에 이르는 클라우드 서비스를 연결 및 전송하는 중개자

5개의 행위자는 서로 상호작용을 한다. 캐리어만 밖에 있음

사용 시나리오 1 : 클라우드 소비자가 제공업체와 직접 접촉하지 않고 브로커에게 서비스를 요청하고 중개업체는 다중 서비스를 조합 또는 현재 서비스 개선 등 새로운 서비스를 제공한다.

사용 시나리오 2 : 클라우드 제공자는 2개의 고유한 SLA에 참여하고 준비



사용 시나리오 3 : 클라우드 서비스 구현의 운영 및 보안에 대한 평가를 수행
클라우드 감사는 클라우드 소비자 및 클라우드 제공자와의 상호 작용이 포함됨




SaaS 소비자 : ERP, Human Resource, Email & Office Productivity
PaaS 소비자 : 응용개발자, 테스터, 배포자, 관리자
IaaS 소비자 : 시스템 개발자, 관리자, IT 관리자

클라우드 제공자의 주요 활동 (배포, 조율, 관리, 보안, 개인정보 보호)

Service Deployment : 서비스 배포

자원 추상화 및 레이어에 서비스 레이어(IaaS, PaaS, SaaS)를 구축

각각의 클라우드 인터페이스를 통해 클라우드 소비자가 클라우드 서비스를 이용하는 계층

자원 추상화 및 제어 계층 : 클라우드 제공자가 소프트웨어 추상화를 통해 물리적 컴퓨팅 자원에 대한 접근을 제공하고 관리하기 위해 사용하는 시스템 구성요소
- 소프트웨어 구성요소 : 하이퍼바이저, 가상머신, 가상 데이터 스토리지, 자원 추상화 및 관리 컴포넌트

물리적 자원 계층 : 물리적 컴퓨팅 자원, 즉 하드웨어 자원
- 컴퓨터(cpu와 메모리)
- 네트워크(라우터, 방화벽, 스위치, 네트워크 링크, 인터페이스)
- 스토리지(하드 디스크)
- 시설 자원 포함 (냉난방), 물리적 플랜트의 기타 요소, 파워

클라우드 서비스 관리 3가지

1. 비즈니스 지원(Business Support) : 고객, 계약, 재고 관리, 회계 & 요금 청구, 보고 & 감사 가격 결정 & 등급 결정
2. 프로비저닝/구성(Provisioning/Configuration) : 신속한 프로비저닝, 자원 변경, 모니터링 & 보고, 사용량 측정, SLA 관리
3. 이식성/호환성(portability/interoperability) : 데이터 이식성(데이터 복제, 대량 데이터 전송), 서비스 호환성(통합 인터페이스), 시스템 이식성(VM 이미지 이동, 앱/서비스 이동)

클라우드 감사 : 클라우드 서비스 제어에 대한 독립적인 검사를 수행
클라우드 서비스, 정보 시스템 운영, 성능, 클라우드 컴퓨팅 보안에 대한 독립적인 평가를 수행

클라우드 감사의 역할 3가지

보안 제어 감사 : 조직 정보 시스템 내에서 사용되는 보호 장치 또는 대책

개인 정보 영향 감사 : Privacy impact : 개인 정보 보호 법률 및 규정을 준수하고 개발 및 운영의 모든 단계에서 개인 정보의 기밀성, 무결성 및 가용성을 보장하는 데 도움

성능 감사 : 클라우드 서비스의 성능 평가

클라우드 중개자 서비스 3가지

중계(Intermediation) : 일부 특정 기능을 개선하거나 가치가 부가된 서비스 제공
통합(Aggregation) : 다중 서비스들을 하나 이상의 신규 서비스로 결합하고 통합
중재(Arbitrage) : 서비스 통합과 유사

클라우드 캐리어 : 클라우드 소비자와 제공자 사이에서 연결성과 전송을 제공하는 중개자 역할
소비자에게 접근 기능을 제공 : 네트워크 등등
클라우드 서비스의 분배, 네트워크 정보통신 캐리어, 전송 에이전트에 의해 제공


소비자와 제공자 사이의 범위(AMO, SPI)

Application Layer -> SaaS 어플리케이션은 SaaS 소비자에게 사용된다.
어플리케이션은 PaaS, IaaS 소비자 그리고 SaaS 제공자에 의해 설치/관리/유지된다.

Middleware Layer -> PaaS 미들웨어는 PaaS 소비자에게 사용된다.
미들웨어는 IaaS 소비자, 숨은 SaaS 소비자, PaaS 제공자에 의해 설치/관리/유지된다.

Operating System Layer -> IaaS 소비자는 geust OS에 대한 모든 책임을 진다.
IaaS 제공자는 host OS를 제어한다. OS, 드라이버 포함 SaaS/PaaS 소비자에게 숨겨진다.

클라우드 제공자와 정부 클라우드 소비자의 책임 공유 -> 물리적 ~응용 보안까지
클라우드 제공자의 보안 의무 : 개인정보 PI, 개인식별 정보 PII

가상화 기술 : 물리적 자원을 가상의 논리적인 자원으로 전환

가상화 층 : VMM 또는 하이퍼바이저
가상 머신 : 추상화 가상 자원, VM 사이 격리(isolation)

가상화 기술의 기본적인 특성

Partitioning 파티셔닝 : 물리적인 한 개의 자원을 논리적으로 분할하여 효율적으로 사용
Encapsulation 캡슐화 : 가상 머신은 캡슐 형태의 폴더와 파일로 존재, 이동 및 복사가 가능
Isolation 격리 : 장애 및 보안에 대비 각 vm은 캡슐홛되어 관리 가능
하드웨어 독립성 : 하드웨어에 독립적으로 가동, vm의 이주 가능
V2P(Virtual machine to Physical machine) P2V 이주 가능

가상화의 핵심 키워드
추상화, 다중화, 격리, 통합(여러 서버를 하나의 강력한 서버 또는 서버 클러스터로 결합하는 프로세스)

서버 통합 Server Consolidation



가상화 SW 
공개 : Xen, VMware Player, 오라클 Virtual Box, 리눅스 KVM
상용 : XenServer, VMware vShpere, Hyper-V

가상화 기술의 장점 (통고집이모) 똥고집 이모..
서버 통합으로 이용률 증가, 관리 비용 절감
수행 환경을 다른 환경들에게서 격리
여러 물리자원들을 단일 가상 자원으로 집단화
시스템의 이동성을 증가 : 응용과 OS 마이그레이션
새 시스템이나 미개발 하드웨어의 모의실험

장점 요약 : 통합, 고립, 집단, 이동, 모의실험

가상화 기술이 클라우드 컴퓨팅에서의 필요성

1. 가상 머신 동적 관리(virtual machine dynamic management)
클라우드는 요구사항 만족 - 즉각적인 배치, 유연한 확장/축소, 백업/다중 인스턴스를 기반으로 한 비즈니스 연속성 지원

2. 하드웨어에 독립적 - 다양한 물리적인 서버에 이동하여 동작이 가능
클라우드 요구조건 만족 - 부하 균등, 확장성, 장애 극복, 회복력

3. 서버 통합
다중의 가상머신을 동시에 구동하여 협업이 가능
-클라우드 요구조건 : 자원 풀링, 요구 기반 자원 할당

4. 경제성
저비용, 하드웨어 이용률 증가, 다운타임 감소, 관리의 효율성
-클라우드 요구조건 : 사용한 만큼 지불

Xen Architecture (반 가상화)


도메인0 : Xen에서 가장 기본적으로 생성되는 컨트롤 도메인 Guest OS
xen이 부팅될때 생성되어 다른 domain들을 부팅/종료하고 domain의 하드웨어 접근 작업을 관할
제노리눅스 환경에서 제어 소프트웨어를 실행
가상 머신 생성, 종료, 자원할당, 컨트롤 플레인 소프트웨어 수행

도메인U : 특권이 없는 나머지 도메인 GuestOS : 생성되는 순서에 따라 이름 부여, DOM0, DOM1, DOM2 ~

Xen 하이퍼바이저는 이를 도메인0에게 전달
도메인0은 받은 요청을 실제 디바이스에 전달

Xen 하이퍼바이저 : 다양한 게스트 OS를 호스팅
가상 머신 : 반가상화 기법이 적용된 GuestOS

Toolstack : 도메인 생성, 소멸, 설정을 위한 인터페이스 제공

컨테이너 : OS의 가상화를 통해 소프트웨어 실행 환경을 제공
GuestOS X 각 컨테이너는 그 자신의 응용 소프트웨어를 포함
동일 시스템 상의 다른 컨테이너들과는 분리된 자원을 이용

컨테이너 데몬 : 컨테이너 연산을 지원 : create, start, kill, delete
가상 머신 방법보다 오버헤드가 적음

컨테이너 이미지 : 주어진 컨테이너 안에서 실행될 수 있는 소프트웨어의 패키지

Docker : 환경에 구애 받지 않고 애플리케이션을 신속하게 배포 및 확장 가능
Docker 컨테이너는 실행중인 다른 컨테이너와 os 커널을 공유한다. 격리 x
컨테이너 가상화 기술로 별도 운영체제를 요구하거나 포함하지 않음
리눅스 커널이 제공하는 기능들인 cgroups, namespace 기법을 사용하여 구현

cgroups : 컨트롤 그룹이라는 새로운 커널 기능, 리소스 사용량 제한
Namespace : 한 프로세스 집합이 한 리소스 집합을 보고 다른 프로세스 집합이 다른 리소스 집합을 보도록 커널 ㅣ소스를 분할

가상화 소프트웨어 구현 방법에 따른 가상화 분류
가상화 sw가 물리 자원을 가상 자원으로 추상화할 때
2가지 상충관계(tradeoff)
- 가상화 소프트웨어가 생성하는 가상 자원은 실제 물리 자원과 동일한 인터페이스를 제공해야함 
- 가상화 소프트웨어는 Guest OS의 동작 제어와 자원 접근을 통제할 수 있어야한다.

가상화 소프트웨어 구현 층에 따른 가상화 분류
하드웨어 기반 가상화 : 물리 자원과 운영체제 사이에 소프트웨어 추상화 계층을 추가
장점 : 메모리 사용량이 적고 가상화에 따른 부하가 적음
단점 : 지원하는 장치가 OS 기반 가상화에 비해 적고 호환성 이슈가 있다.

운영체제 기반 가상화 : OS 내부에서 가상 자원들을 생성하고 이 자원에서 Guest OS가 동작
장점 : Host OS가 이미 지원하는 장치를 모두 활용 가능
단점 : 가상화 층의 도입에 따른 성능 저하 발생

미들웨어 기반 가상화 : 프로그래밍 언어 또는 런타임 라이브러리 수준에서 가상머신을 제공
시스템의 이용률을 향상시키려기 보다 이식성과 상호 호환성을 향상시키려는 목적 

네트워크 가상화 : 하드웨어로 제공되던 리소스를 소프트웨어로 추상화
(VPN, VLAN, NFV, SDN)

VPN : 가상레이어로 네트워크를 독립된 가상 사설 네트워크로 분할
사설 네트워크, 공증네트워크 : 공인IP

VLAN : LAN의 가상화 하나의 물리네트워크를 여러 개의 논리 네트워크로 분할

하드웨어 요소/기능을 추상화(예 : 연결, 스위치, 라우터 등)

NFV (네트워크 기능 가상화 Network Function Virtualization)
특정 네트워크 기능(방화벽, 로드밸런서)을 제공하는 하나 이상의 하드웨어 장치를 가상화

SDN (소프트웨어 정의형 네트워킹 Software Definition Networking)
네트워크 트래픽 라우팅을 제어하는 H/W를 가상화, 장치의 제어부와 전송부를 분리

스토리지 가상화(RAID, SAN, NAS)

RAID : Redundant Arrays of Independent Disk
RAID 0 : 블록 스트라이핑
RAID 1 : 단순 중복 저장 기능 Mirroring
RAID 2 : 2비트 이상의 오류 검출 비트 추가
RAID 3 : 패리티 비트 추가
RAID 4 : 0 ~ 3을 합쳐서 성능과 확실성 확보

NAS (Network Attached Storage) 네트워크 결합 스토리지
파일 수준 저장 방법으로 서로 다른 네트워크 클라이언트에 데이터 접근 권한을 제공한다
직접 연결하지 않고 근거리 LAN 수준의 네트워크를 제공 
파일공유, 동시 접속이 필요한 업무에 사용

<-> 반대 개념은 컴퓨터에 직접 연결해서 쓰는 DAS

SAN (Storage Area Networks) : 스토리지 에어리어 네트워크
대규모 네트워크 사용자를 위한 LAN선 대신에 광케이블을 사용하는 고속 네트워크 시스템
장점 : 고성능 고용량 업무에 사용 단점 : 비쌈

분산 처리 기술 (GFS, MapReduce, HaDoop)

GFS : 구글 파일 시스템
Big Table 빅테이블 : 열 지향 키-값 구조 키/값 매핑으로 구성
구조적 데이터 처리를 위한 분산 저장 시스템
MapReduce : 대용량 데이터 처리, 분산-병렬 데이터 처리
한명이 4주 작업할 일을 4명이 나누어 1주일에 끝내는 것
맵리듀스 과정
Input - Splitting -> Mapping -> Suffling -> Reducing -> Final Result

다중화 대상: 서버, 로드밸런서, 네트워크 장치 : 장애 발생시 예비 장비로 장애 극복

- 네트워크 허브 고장
- 무중단 -고가용성 서비스 구현을 위함

퍼블릭 또는 프라이빗으로 분류

서브넷의 CIDR 블록은 중첩될 수 없다.

서브넷 수준에서 퍼블릭 IP주소 자동 할당 설정을 통해 자동으로 할당

탄력적 네트워크 인터페이스
인스턴스에 연결, 인스턴스에 분리하고 다른 인스턴스에 연결하여 네트워크 트래픽 리디렉션 기능을 지원
VPC의 각 인스턴스마다 기본 네트워크 인터페이스가 있어서 VPC 주소 범위에 속하는 프라이빗 IPv4 주소가 탄력적 네트워크 인터페이스에 할당됨

각 서브넷의 기본 네트워크 인터페이스의 프라이빗 주소가 탄력적 네트워크 인터페이스에 할당됨

각 서브넷은 라우팅 테이블과 연결되어야 합니다. (최대 1개)

아웃 바인드 될때 인터넷과 연결 가능


NAT(Network Adress Translation) : 서브넷 안에 NAT 게이트웨이가 퍼블릭과 프라이빗 라우팅 테이블에 연결되어 있음 

NAT 게이트웨이는 퍼블릭, 프라이빗 서브넷에 있으면서 퍼블릭, 프라이빗 라우팅 테이블 모두와 연결 가능

VPC 피어링 : VPC 공유 AWS 리전 간에 연결할 수 있습니다.

AWS 사이트 간 VPN은 프라이빗 서브넷의 라우팅 테이블에 가상 게이트웨이를 연결하여
사이트 간 VPN 연결을 가능하도록 합니다.





VPN을 사용하지 않고 AWS Direct Connect를 사용하여 직접 연결 할 수 있습니다.

보안 그룹은 인스턴스 수준(서브넷 보다 더 작은 범위)에서 작동합니다.

보안 그룹 : 인바운드 및 아웃바운드 인스턴스 트래픽을 제어하는 규칙이 있습니다.
기본 보안 그룹은 모든 인바운드 트래픽을 거부하고 모든 아웃바인드 트래픽을 허용합니다.
보안 그룹은 상태를 저장합니다.

모든 들어오는 것은 거부 나가는 트래픽은 허용

사용자 보안 그룹은 허용 규칙을 지정할 수 있지만 거부 규칙은 지정할 수 없습니다.
모든 규칙은 트래픽 허용을 결정하기 전에 평가됩니다.

어떤 것을 거부할 것인지는 지정할 수 없습니다.

네트워크 ACL은 서브넷 수준에서 작동합니다.

네트워크 ACL은 별개의 인바운드 및 아웃바인드 규칙이 있고 각 규칙은 트래픽을 허용하거나 거부할 수 있습니다.
기본 네트워크 ACL은 인바운드 및 아웃바운드 트래픽을 모두 허용합니다.
네트워크 ACL은 상태를 저장하지 않습니다.

사용자 지정 네트워크 ACL은 규칙을 추가하기전에 모든 인바운드, 아웃트래픽 거부
허용 및 거부 규칙 모두를 지정할 수 있습니다.
